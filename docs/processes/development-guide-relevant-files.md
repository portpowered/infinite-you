# Development Guide Relevant Files

This inventory records the checked-in files and directories that the maintainer development guide should describe when it gives repository-root workflow instructions.

| Path | Role | Notes |
| --- | --- | --- |
| `go.mod` | Repository-root marker | Maintainer commands and worktree-aware tests should treat the directory containing `go.mod` as the canonical repository root. |
| `Makefile` | Root command surface | The development guide should describe quality and generation commands as root-level invocations instead of teaching a nested package workflow. |
| `api/` | Authored API contract workspace | OpenAPI validation and bundling start from the repository root, then shell into `api/` only where the documented workflow requires it. |
| `api/components/schemas/shared/` | Shared OpenAPI fragment family | Cross-surface helper schemas such as maps, diagnostics helpers, and resource helper types live here and referenced fragments must use file-relative `$ref`s. |
| `api/components/schemas/api/` | API payload OpenAPI fragment family | Request, response, status, token, and pagination contract fragments are authored here one schema per file. |
| `api/components/schemas/data-models/` | Data-model OpenAPI fragment family | Public named-factory, topology, work, relation, and other reusable base model schemas are authored here and should stay split from `api/openapi-main.yaml`. |
| `api/components/schemas/factory-world/` | Additive factory-world OpenAPI fragment family | Dashboard-facing read-model schemas belong here one schema per file rather than remaining inline in `api/openapi-main.yaml`. |
| `cmd/factory/` | CLI entrypoint | Root-level build and smoke commands compile or execute the `factory` binary from this source tree. |
| `docs/development/*-closeout.md` | Cleanup verification artifacts | Narrow cleanup lanes record the exact root-level validation bundle here when maintainers need durable proof beyond `progress.txt`. |
| `docs/development/openapi-schema-standardization-inventory.md` | OpenAPI cleanup inventory | Records the authored fragment layout, remaining inline schemas, and the canonical bundle and generation verification surfaces for schema-standardization work. |
| `docs/development/root-factory-artifact-contract-inventory.md` | Root artifact contract inventory doc | The checked-in root artifact table must stay in lockstep with the enforced entries in `internal/testpath/artifact_contract.go`. |
| `docs/development/development.md` | Active maintainer guide | Must describe the real repository-root layout used in this checkout and avoid stale `libraries/agent-factory` instructions. |
| `.github/workflows/ci.yml` | Repository CI contract | Contributor docs should name this workflow and mirror its root-level validation commands and stated non-deployment scope. |
| `internal/testpath/artifact_contract.go` | Enforced root artifact contract | Root-level factory artifact additions, removals, and redirect stubs are test-enforced here and must stay synchronized with the inventory doc. |
| `internal/contractguard/functional_layout_test.go` | Functional decomposition guard surface | Repository-owned layout enforcement for the decomposed functional suite lives here; it keeps `functionallong` files out of the default lane unless they sit under `tests/functional/...` with `_long_test.go` names, and it freezes new helper shims in `tests/functional_test/`. |
| `factory/` | Maintainer workflow surface | Contains checked-in operator guidance and active inbox directories that the development guide may reference for workflow-related tasks. |
| `pkg/api/handlers.go` | Handwritten API decode and validation boundary | Unsupported or legacy public request fields for `POST /work` still need explicit raw-JSON rejection here because generated request structs do not reject unknown keys by themselves. |
| `pkg/api/openapi_contract_test.go` | OpenAPI contract guard surface | Focused authored-versus-bundled contract assertions live here, including fragment-layout and `/events` schema wiring checks. |
| `pkg/api/testdata/canonical-event-vocabulary-stream.json` | OpenAPI vocabulary fixture | Canonical bundled-contract fixtures for event payload validation live here and must be updated alongside public schema field renames. |
| `pkg/interfaces/public_factory_enums.go` | Shared public factory enum normalizers | Shared worker and workstation alias ownership lives here; exported callers must choose explicit strict or permissive helpers instead of duplicating alias tables. |
| `pkg/interfaces/factory_runtime.go` | Handwritten public work-request boundary structs | Watched-file batch ingestion, generated worker output parsing, and fixture helpers still marshal these structs directly, so their JSON tags must stay aligned with the camelCase OpenAPI contract. |
| `pkg/config/factory_config_mapping.go` | Generated factory-config boundary alias rejection owner | Mirrored legacy alias guards for top-level objects and nested `definition` objects live here; keep one field inventory per boundary type and vary only the path passed into the shared rejection helper. |
| `pkg/config/factory_config_mapping_test.go` | Generated factory-config boundary regression surface | Boundary rejection regressions should stay focused here by asserting `generatedFactoryBoundaryErrorPrefix` plus the full path-specific retirement guidance string for top-level, nested `definition`, and nested `definition.cron` aliases. |
| `pkg/workers/provider_behavior.go` | Provider CLI behavior owner | Provider-specific argument construction and command-request shaping live here; worker inference paths should call this seam instead of reintroducing forwarding wrappers in `inference_provider.go`. |
| `pkg/` | Go implementation surface | Package-specific test commands in the guide should reference the real package paths under this root. |
| `tests/` | Smoke and fixture surface | Functional and release-facing checks run from the repository root against these checked-in fixtures. |
| `tests/functional/` | Behavior-first functional package tree | New functional decomposition work should land here under the documented package taxonomy, using `make test-functional` for the default non-long lane and `make test-functional-long` for the opt-in slow lane. |
| `tests/functional/internal/support/` | Shared functional helper seam | Cross-package functional harnesses, fixture lookup helpers, command-runner helpers, and reusable assertions should move here once more than one behavior package needs them. |
| `tests/functional/README.md` | Functional package and command contract | Defines package ownership, transport prefix rules, long-lane placement, and temporary coexistence with `tests/functional_test/` during migration. |
| `tests/functional_test/` | Behavioral regression suite | Functional tests should prove observable runtime, API, CLI, dashboard, or emitted-event behavior; source scans, docs topology checks, bundle internals, and command or route inventories are maintenance guards rather than product-facing smoke coverage. |
| `ui/` | Embedded dashboard workspace | UI build, test, and Storybook commands remain part of the same repository-root workflow. |
| `ui/src/testing/replay-fixture-catalog.ts` | Replay integration test contract | Browser-backed dashboard smoke coverage should register scenario metadata here so coverage reporting and integration assertions stay on one source of truth. |
| `ui/scripts/write-replay-coverage-report.ts` | Replay coverage reporter | Package scripts should use this repository-owned reporter to validate replay metadata instead of embedding ad hoc fixture maps in tests or CI. |
| `ui/scripts/normalize-dist-output.mjs` | Embedded asset normalizer | The documented UI build path ends by normalizing Vite output names and refreshing `ui/dist_stamp.go` so committed embed assets stay stable for Go builds and CI diffs. |

## Reusable Rules

- When maintainer docs describe command execution, anchor the instructions to the repository root that contains `go.mod` and `Makefile`.
- If a workflow temporarily changes directories, state that it starts from the repository root and why the subdirectory hop is required.
- When GitHub Actions or other automation is added, prefer repository-owned root commands or package scripts that the maintainer guide already documents instead of inventing CI-only command sequences.
- When contributor docs mention the repository CI workflow, mirror the exact root-level command sequence and its stated scope from `.github/workflows/ci.yml` so local reproduction and review expectations do not drift.
- When UI assets are committed for Go embedding, keep the build pipeline responsible for normalizing output filenames and refreshing any cache-busting stamp files instead of hand-editing `ui/dist/`.
- When browser-backed UI replay tests and replay coverage reports share the same scenarios, keep that metadata in one repository-owned catalog so the tests, scripts, and docs cannot silently drift.
- When GitHub Actions uses `actions/setup-go` against a module that declares a newer `toolchain` than its base `go` version, prefer uncached setup unless you have verified that cache restore does not collide with the auto-downloaded toolchain files in later jobs.
- When a cleanup lane closes with path or contract-alignment work, record the exact root-level verification commands in a `docs/development/*-closeout.md` artifact so the proof survives beyond `progress.txt`.
- When functional tests need `os.Chdir`, route the directory swap through the shared helper and serialize it with a package-level lock; process working directory is global state and will race any `t.Parallel()` coverage otherwise. If the same test changes directories more than once or does so inside subtests, use the scoped helper form so the lock is released between command executions instead of being held until parent-test cleanup.
- The shared working-directory helpers for decomposed functional packages live in `tests/functional/internal/support/workdir.go`; use `support.SetWorkingDirectory(...)` for one-shot test setup, `support.WithWorkingDirectory(...)` when the same test needs multiple scoped directory swaps, and `support.ResolvedRuntimePath(...)` instead of reintroducing fresh `os.Chdir` helpers in each behavior package.
- Put new decomposed functional coverage under `tests/functional/`, choose the behavior package before the filename prefix, and reserve `tests/functional_test/` for temporary migration compatibility until the legacy bucket is retired.
- When functional helpers are reused across behavior packages, promote them into `tests/functional/internal/support` instead of importing, copying, or extending another package's `*_test.go` helpers.
- When migrating a legacy functional test that still depends on `tests/functional_test/testdata`, reach that fixture through `support.LegacyFixtureDir(...)` and move any newly shared provider or assertion helpers into `tests/functional/internal/support` instead of reintroducing local compatibility shims.
- Replay-owned fixture, serialization, and divergence coverage belongs in `tests/functional/replay_contracts`; keep the package on `support.LegacyFixtureDir(...)` while fixtures are still rooted under `tests/functional_test/testdata` instead of leaving those replay assertions behind in the legacy bucket.
- Script request/response canonical-history and recorded-artifact boundary assertions belong in `tests/functional/replay_contracts` even when the exercised runtime path is a script worker; the owning behavior is event-history and replay-contract stability, not generic executor mechanics.
- Command-request payload preservation and thin recorded-artifact dispatch correlation checks belong in `tests/functional/replay_contracts` when they prove replay-visible event/request contracts rather than live HTTP/runtime projections; keep any surviving `/events` mirror smoke in `tests/functional_test/` only until its server harness migrates.
- When a replay-owned smoke still needs to prove that a replay artifact mirrors the live `/events` stream and dashboard/world-state projections, keep that observable contract in `tests/functional/replay_contracts` and add only the minimal package-local service-backed helper needed there instead of restoring the test to `tests/functional_test/` or treating it as generic runtime API ownership.
- Runtime/API transport-alignment checks such as watched-file versus HTTP request normalization or file/HTTP/replay generated-schema parity belong in `tests/functional/runtime_api`, while public worker payload and replay-artifact visibility contracts belong in `tests/functional/replay_contracts`.
- Canonical flatten/replay/runtime-config alignment and retired-alias rejection coverage also belongs in `tests/functional/runtime_api` when the observable contract is that authored config, generated schema, replay loading, live `/events`, and runtime execution stay aligned at the public runtime/API boundary.
- Live `/events` stream timeline, dashboard/world-view projection, and selected-tick reconstruction smokes belong in `tests/functional/runtime_api` when they prove the running HTTP/runtime boundary; keep artifact-format conversion and replay-regression ownership in `tests/functional/replay_contracts`.
- Inference-attempt history, retry-stream correlation, and thinned inference dispatch view reconstruction belong in `tests/functional/runtime_api` when the assertions are about live `/events`, dashboard/session read models, or workstation-request projections rather than replay artifact format guarantees.
- Script executor command-shaping, workstation template resolution, provider CLI prompt/worktree assembly, and runtime workdir/env propagation assertions belong in `tests/functional/providers`; the owning behavior is worker/provider execution at the subprocess boundary, not generic workflow narration or live HTTP runtime projection.
- Runtime log rolling-policy, command-output redaction, and replayed command-diagnostics assertions also belong in `tests/functional/providers`; the owning behavior is subprocess execution observability at the worker/provider boundary, not generic runtime API projection.
- Provider error normalization, lane-isolation execution checks, retryable-failure metadata, and guarded provider loop-breaker assertions belong in `tests/functional/providers`, while runtime snapshot and dashboard throttle-pause projection smokes belong in `tests/functional/runtime_api`; the owning behavior is either provider/subprocess execution or the live runtime/API boundary.
- When multiple decomposed behavior packages need provider error corpus fixtures, load them through `tests/functional/internal/support.ProviderErrorCorpusEntry(...)` instead of recreating per-package corpus-loader helpers.
- Mock-worker accept/reject/script execution and service command-runner completion coverage also belongs in `tests/functional/providers`; the owning behavior is execution-path substitution at the worker/provider boundary, not generic smoke or workflow progression.
- Service-harness mock-worker/custom-executor coverage belongs in `tests/functional/smoke`, while HTTP functional-server compatibility and provider-override regressions belong in `tests/functional/runtime_api`; the owning behavior is either fast harness smoke or the live runtime/API boundary, not the shrinking legacy mixed bucket.
- Service-harness command-runner override alignment also belongs in `tests/functional/smoke`, while the same override contract through the live functional HTTP server belongs in `tests/functional/runtime_api`; the owning behavior is either local harness execution or the runtime/API boundary where the override must remain observable.
- Public dashboard shell, `/status`, and canonical `/events` history smokes belong in `tests/functional/runtime_api` when they verify the live runtime/API boundary rather than replay artifact ownership.
- First-run HTTP API smokes that prove preseeded work reaches terminal completion and `/status` stays queryable belong in `tests/functional/runtime_api`; the owning behavior is the live runtime/API boundary rather than generic smoke or bootstrap scaffolding.
- Fast user-visible command availability checks such as `agent-factory docs` packaged-topic smoke, and loaded-config execution smokes that only prove default configuration still runs, belong in `tests/functional/smoke` rather than staying in the shrinking legacy mixed bucket.
- Service-harness end-to-end dispatch completion smokes also belong in `tests/functional/smoke`; when the assertion is simple completion confidence for one or many seeded work items, keep it with the fast harness-owned coverage instead of the legacy mixed bucket.
- Simple stateless collector pipeline and multi-item execution smokes also belong in `tests/functional/smoke`; the owning behavior is fast service-harness execution confidence for basic multi-step completion rather than scheduler, replay, or runtime API projection.
- When moving a test file out of `tests/functional_test/`, keep any temporary legacy compatibility shim as small and helper-only as possible so still-unmigrated callers compile without pulling the full behavior back into the mixed legacy bucket.
- When migrating a legacy functional test off a helper that still serves unmigrated callers, reuse or extend an existing allowlisted helper file in `tests/functional_test/` instead of adding a brand-new `*_helpers_test.go` shim; `internal/contractguard/functional_layout_test.go` rejects new legacy helper files even when the seam is temporarily necessary.
- When decomposed functional packages need generic harness polling, token-count, or provider-call helpers across behavior areas, promote those seams into `tests/functional/internal/support`; keep service-backed HTTP API helpers package-local to `tests/functional/runtime_api` until another behavior package proves they are truly shared.
- When a decomposed functional package needs a synthetic `factory.json` fixture rather than a copied checked-in fixture, build it through `tests/functional/internal/support.ScaffoldFactory(...)` and the other support writers instead of reviving `scaffoldFactory(...)` in `tests/functional_test/` or cloning ad hoc filesystem setup in each behavior package.
- Runtime dispatch-timing and in-flight read-model coverage belongs in `tests/functional/runtime_api`; when those tests only need token identity summaries, prefer `tests/functional/internal/support.DeriveTokenIdentities(...)` over keeping a duplicate `token_identity_helpers_test.go` seam in `tests/functional_test/`.
- Work-in-queue scheduler priority, cron ordering, and runtime-config scheduler wiring coverage belongs in `tests/functional/runtime_api`; the owning behavior is live runtime scheduling and replay-visible dispatch ordering rather than generic smoke or guard semantics.
- When a slow functional scenario still belongs to a decomposed behavior package, gate it with a package-local `//go:build functionallong` test file and keep `make test-functional` package-discovery-only instead of adding manual excludes or pulling `tests/functional_test/` back into the default lane.
- When a functional file joins the opt-in slow lane, name it `*_long_test.go` and keep it under `tests/functional/<behavior-package>/`; `internal/contractguard/functional_layout_test.go` enforces both the build tag and the placement.
- When a legacy functional test loops over many unrelated fixtures just to prove they load, replace that umbrella sweep with behavior-package tests that assert each flow's observable terminal tokens or provider interactions, then delete or long-gate the mixed sweep instead of migrating it wholesale.
- When a watcher-driven functional test is really proving guard or batch fan-in semantics, keep it in the owning behavior package such as `tests/functional/guards_batch` and use a `watcher_` filename prefix for discoverability; only pure watcher submission-flow coverage should live with the workflow-oriented packages.
- When a watched-file scenario submits a parent/child batch, keep it in `tests/functional/guards_batch` even though the transport is a watcher input; the owning behavior is relation fan-in and parent failure routing, not generic file pickup.
- Idea-to-PRD promotion, batch or resource-limited ideation pipelines, planner/script/executor/reviewer limit loops, and dispatcher lifecycle lineage coverage belong in `tests/functional/workflow` because the owning behavior is cross-work-type workflow progression rather than provider, replay, or runtime API plumbing.
- Ralph scaffold request intake, generated `prd.md`/`prd.json`/`progress.txt` artifact handoff, and one-story-per-iteration execution-loop coverage belong in `tests/functional/workflow`; the owning behavior is generated workflow progression across planner and executor workstations rather than generic integration smoke.
- Executor input-token payload/tag propagation, reviewer rejection-feedback loops, and parent-child or depends-on lineage preservation for the `code_review` flow belong in `tests/functional/workflow`; the owning behavior is workflow progression and feedback routing rather than generic provider mechanics.
- Workstation stop-word routing, workstation-level stop-word overrides over worker stop tokens, and equivalent acceptance-versus-failure progression checks belong in `tests/functional/workflow`; the owning behavior is workstation progression semantics through the workflow graph rather than provider transport or runtime API projection.
- Workflow-version reloads, rejection-loop regressions, and independent config-evolution assertions also belong in `tests/functional/workflow`; the owning behavior is workflow progression across authored factory versions rather than generic smoke or provider execution.
- Multi-output lineage/color propagation, logical-move passthrough semantics, repeater progression, unresolved runtime-template routing, and input-name propagation all belong in `tests/functional/workflow`; the owning behavior is workflow progression and token-shaping through ordinary workstation chains rather than provider-only execution or runtime API read models.
- Failure-arc fallback, failed-token immutability, and dependency-cascade routing belong in `tests/functional/guards_batch` because the owning behavior is batch/guard dispatch semantics rather than generic workflow narration.
- Canonical request-batch normalization, dependency enforcement, chained batch fan-in, and batch submission rejection coverage belong in `tests/functional/guards_batch`; when those tests need to mutate a legacy factory fixture or workstation config during migration, use `tests/functional/internal/support.UpdateFactoryConfig(...)` and `support.WriteWorkstationConfig(...)` instead of recreating package-local filesystem helpers in the shrinking legacy bucket.
- Current-factory activation, named-factory export/import, flatten/expand standalone config round-trips, and portable authored-vs-canonical layout assertions belong in `tests/functional/bootstrap_portability` even when they exercise `POST /factory`, `GET /factory/~current`, watcher activation, persisted layout seams, or config CLI commands; the owning behavior is customer-facing factory lifecycle and portability rather than generic runtime API coverage.
- Invalid worker-reference validation smokes also belong in `tests/functional/bootstrap_portability`; the owning behavior is factory bootstrap rejecting broken authored configuration before runtime startup, not generic runtime API or provider execution coverage.
- Treat `tests/functional_test/` as a shrinking compatibility surface only. If a new helper would be shared across behavior packages, move it into `tests/functional/internal/support` instead of adding another `*_helpers_test.go` or `*_compat*_test.go` file in the legacy bucket.
- Keep `tests/functional_test/` focused on observable behavior. If a check only proves source layout, docs linkage, asset bundle contents, or command or route registration, move it to a narrower guard surface instead of adding it to the functional suite.
- When a functional test covers `agent-factory docs`, run the command from a temp working directory without a local docs tree and assert user-visible headings or stable content markers rather than byte-for-byte parity with packaged markdown files.
- Stateless fixture execution, thin dispatch lookup references, and workstation-to-worker resolution smokes belong in `tests/functional/smoke`; the owning behavior is fast service-harness execution confidence rather than bootstrap portability or runtime API projection.
- When `make lint` fails in `cmd/deadcodecheck`, remove orphaned `_test.go` helper files and unused test wrappers before touching `docs/development/deadcode-baseline.txt`; the baseline is for accepted remaining debt, not abandoned local helpers.
- Use `pkg/api/openapi_contract_test.go` for narrow OpenAPI contract guards when the work is about authored schema structure or bundled route/schema alignment rather than handler runtime behavior.
- Keep `docs/development/root-factory-artifact-contract-inventory.md` and `internal/testpath/artifact_contract.go` synchronized; the doc is not descriptive-only, it is a checked-in contract surface with order-sensitive tests.
- When public OpenAPI field names change, update `pkg/api/testdata/canonical-event-vocabulary-stream.json` together with the contract guards so fixture validation keeps exercising the current bundled vocabulary.
- When public request-batch field names change, update `pkg/interfaces/factory_runtime.go`, `pkg/factory/work_request_json.go`, and watched-file/worker batch fixtures together; those handwritten JSON boundaries are not generated and will drift silently if only `api/openapi.yaml` and generated clients are regenerated.
- Keep shared public worker and workstation enum aliases in `pkg/interfaces/public_factory_enums.go`; generated/OpenAPI boundary normalizers should call the strict helpers while generated/public round-trip helpers should call the permissive ones instead of cloning alias tables in `pkg/config` or UI export code.
- In `pkg/config/factory_config_mapping.go`, keep mirrored generated-boundary retirement guards data-driven: define one retired-field inventory per boundary type and reuse it for both the top-level object and any nested `definition` object so only the caller-owned path text differs.
- `POST /work` is the single-submit surface and should reject relation graphs explicitly; named relation wiring belongs on `PUT /work-requests/{requestId}`, where the handler can resolve `sourceWorkName` and `targetWorkName` safely.
- `decodeSubmitWorkRequestBody` is the handwritten validation seam for `POST /work`; use the raw JSON field map there whenever generated structs need compatibility checks beyond Go's default `json.Unmarshal` behavior, especially when unary submit fields differ from canonical batch fields.
- For Redocly-bundled vendor extensions that must keep a schema pointer in `api/openapi.yaml`, store the extension value as a string JSON Pointer; nested `$ref` objects under `x-*` fields are inlined during bundling.
- When dashboard or world-view code needs runtime-only observability that is not reconstructed from canonical events, keep the event reducer unchanged and overlay the live engine snapshot fields at the projection boundary that already owns both inputs.
- For dispatcher pause-gating tests, use scheduler doubles that derive decisions from the `enabled` slice they receive so the assertions reflect the real FIFO/work-queue scheduler contract instead of depending on impossible post-selection decisions.
- When `make lint` fails on `cmd/deadcodecheck`, remove truly stale symbols first and only then refresh `docs/development/deadcode-baseline.txt` for the remaining accepted library or test-helper debt in the same review.
- Keep provider-specific CLI argument assembly and command-request shaping in `pkg/workers/provider_behavior.go`; `pkg/workers/inference_provider.go` should assemble execution around that seam rather than growing duplicate forwarding helpers.
- Keep provider CLI build-args regression tests adjacent to `pkg/workers/provider_behavior.go`; `inference_provider` tests should stay focused on command assembly and execution behavior instead of duplicating provider flag assertions through wrapper-oriented seams.
- When `inference_provider` tests need to verify provider command payloads, derive the expected args and command request from `providerBehaviorFor(...).BuildArgs(...)` and `BuildCommandRequest(...)` so the regression fails if orchestration drifts away from the canonical provider-behavior owner.
