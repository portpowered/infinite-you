# Development Guide Relevant Files

This inventory records the checked-in files and directories that the maintainer development guide should describe when it gives repository-root workflow instructions.

| Path | Role | Notes |
| --- | --- | --- |
| `go.mod` | Repository-root marker | Maintainer commands and worktree-aware tests should treat the directory containing `go.mod` as the canonical repository root. |
| `Makefile` | Root command surface | The development guide should describe quality and generation commands as root-level invocations instead of teaching a nested package workflow. |
| `cmd/releaseprep/` | Release-preparation entrypoint | The maintainer `make release VERSION=vX.Y.Z` flow shells through this command so git guardrails and readiness sequencing stay cross-platform and testable instead of being duplicated in shell-only Makefile logic. |
| `internal/releaseprep/` | Release-preparation policy logic | Branch, clean-tree, semver-tag, local/remote tag, and readiness-target enforcement lives here and should stay aligned with the maintainer release policy guide. |
| `cmd/releasetagcheck/` | Workflow-facing release-tag entrypoint | GitHub Actions should call this command when validating an explicit release tag or resolving the one semver tag that points at a verified commit. |
| `internal/releasetag/` | Shared release-tag policy helpers | Keep semver tag validation and filtering here so maintainer commands, workflows, and tests enforce one release-tag contract. |
| `cmd/releasesmoke/` | Release smoke entrypoint | Candidate-artifact and released-artifact verification should invoke this repo-owned smoke boundary so packaged-binary checks stay centralized instead of being reimplemented inline in workflow YAML. |
| `internal/releasesmoke/` | Release artifact smoke logic | Packaged-binary startup, `/events` validation, browser-backed dashboard rendering checks, and completed-work verification live here so scripts and GitHub Actions share one observable runtime contract. |
| `.goreleaser.yml` | Release packaging contract | The tag-driven CLI artifact matrix, archive formats, checksum naming, and snapshot-capable packaging flow should stay centralized here instead of being duplicated inline in release workflow YAML. |
| `install.sh` | Repo-owned CLI installer boundary | Consumer fallback installation should derive the same archive and checksum names as `.goreleaser.yml`, resolve the tagged GitHub release surface, and keep platform selection plus checksum verification in one reusable script instead of duplicating that logic across docs or workflow YAML. |
| `scripts/release/` | Cross-platform release smoke wrappers | Shell and PowerShell release smoke entrypoints should stay thin wrappers over the repo-owned Go smoke command or hosted-installer smoke helper instead of drifting into separate platform-specific verification logic. |
| `ui/scripts/release-dashboard-smoke.mjs` | Browser-backed release dashboard verifier | Rendered dashboard smoke should stay in this one Playwright entrypoint so packaged-binary verification proves embedded asset loading plus live dashboard state without duplicating browser assertions in Go tests or workflow YAML. |

| `api/components/schemas/api/` | API payload OpenAPI fragment family | Request, response, status, token, and pagination contract fragments are authored here one schema per file. |
| `api/components/schemas/data-models/` | Data-model OpenAPI fragment family | Public named-factory, topology, work, relation, and other reusable base model schemas are authored here and should stay split from `api/openapi-main.yaml`. |
| `api/components/schemas/factory-world/` | Additive factory-world OpenAPI fragment family | Dashboard-facing read-model schemas belong here one schema per file rather than remaining inline in `api/openapi-main.yaml`. |
| `cmd/factory/` | CLI entrypoint | Root-level build and smoke commands compile or execute the `factory` binary from this source tree. |
| `README.md` | Consumer install and quickstart contract | Consumer-facing install guidance should keep the exact Homebrew, hosted `install.sh`, `go install`, and manual release-archive fallback commands aligned with the repo-owned release surfaces instead of inventing parallel install instructions in one-off docs. |
| `tests/release/release_smoke_test.go` | Release install-surface smoke contract | Packaged-binary runtime checks, the clean-`GOBIN` `go install ./cmd/factory` smoke, and the opt-in published-module `go install` smoke should stay together here so release-oriented install verification remains observable and repo-owned. |

| `docs/development/openapi-schema-standardization-inventory.md` | OpenAPI cleanup inventory | Records the authored fragment layout, remaining inline schemas, and the canonical bundle and generation verification surfaces for schema-standardization work. |
| `docs/development/root-factory-artifact-contract-inventory.md` | Root artifact contract inventory doc | The checked-in root artifact table must stay in lockstep with the enforced entries in `internal/testpath/artifact_contract.go`. |
| `docs/development/development.md` | Active maintainer guide | Must describe the real repository-root layout used in this checkout and avoid stale `libraries/agent-factory` instructions. |

## Reusable Rules

- When maintainer docs describe command execution, anchor the instructions to the repository root that contains `go.mod` and `Makefile`.
- If a workflow temporarily changes directories, state that it starts from the repository root and why the subdirectory hop is required.
- When GitHub Actions or other automation is added, prefer repository-owned root commands or package scripts that the maintainer guide already documents instead of inventing CI-only command sequences.
- When the CLI docs command surface changes, update `README.md` in the same pass and keep its command examples aligned with the guard assertions in `pkg/cli/root_test.go`.
- When contributor docs mention the repository CI workflow, mirror the exact root-level command sequence and its stated scope from `.github/workflows/ci.yml` so local reproduction and review expectations do not drift.
- When the repository adds or changes release automation, keep the workflow trigger model and maintainer commands aligned with `docs/guides/cli-release-policy.md` so tags, documentation, and GitHub Actions do not describe conflicting release paths.
- Keep maintainer install-surface responsibilities in `docs/guides/cli-release-policy.md` aligned with `.goreleaser.yml` and `.github/workflows/release.yml`, including the Homebrew tap secret name, the generated `Casks/agent-factory.rb` location, and the hosted GitHub release `install.sh` asset path.
- Keep maintainer release-preparation git checks in `internal/releaseprep/` and call them through `cmd/releaseprep/` from `Makefile` rather than rebuilding the same branch, clean-tree, or tag-existence logic inline in shell recipes.
- Keep `ui-deps` ahead of `typecheck` in the release-preparation readiness order because the documented repository baseline installs dashboard dependencies before running the Bun-backed UI typecheck on a clean checkout.
- Keep the CLI release target matrix, archive naming, and checksum contract in `.goreleaser.yml`; release workflow YAML should call that shared packaging file rather than re-declaring per-platform artifact details inline.
- Keep consumer install-surface archive resolution and checksum verification in the repo-owned `install.sh`; future release automation should host or invoke that script rather than reimplementing per-platform asset selection in docs or GitHub workflow steps.
- Keep Homebrew cask tap ownership, tagged-release asset URLs, and unsigned-macOS caveat or hook behavior in `.goreleaser.yml` so package-manager install metadata stays coupled to the same release artifacts and checksum contract as the archives.
- When a workflow needs published GoReleaser asset filenames, derive them from the semver version string without the leading `v` because `.goreleaser.yml` archive and checksum names use `{{ .Version }}` rather than the raw tag text.
- Keep packaged-binary release smoke behavior in `internal/releasesmoke/` and invoke it through `cmd/releasesmoke/` plus the thin wrappers in `scripts/release/`; candidate and post-publish workflows should share the same runtime verification contract instead of growing platform-specific shell logic.
- Keep hosted `install.sh` smoke in `scripts/release/smoke-install.sh` and run it from its own release-workflow job, so Linux or macOS installer failures stay classified separately from Homebrew publication, packaged release assets, and `go install` verification.
- Keep the release workflow's Homebrew verification job checking the generated tap cask against the just-published tag's checksum file before install, so stale `Casks/agent-factory.rb` metadata fails with a release-alignment error instead of passing on an older healthy artifact.
- When hosted installer smoke needs repo-local or fake-release coverage, pass `AGENT_FACTORY_INSTALL_*` overrides through `scripts/release/smoke-install.sh` instead of forking `install.sh` or hardcoding GitHub-only URLs into behavioral tests.
- Keep release-oriented `go install` verification in `tests/release/release_smoke_test.go` with a clean temp `GOBIN`; keep the fast repo-owned `./cmd/factory` smoke there for local regression coverage and let the release workflow opt into the published-module `@latest` smoke after tag publication.
- Keep the supported public `go install` command aligned across `go.mod`, `README.md`, `docs/guides/cli-release-policy.md`, and `tests/release/release_smoke_test.go`; if the module path changes in one place and not the others, the documented install surface stops being real.
- Keep browser-backed release dashboard assertions in `ui/scripts/release-dashboard-smoke.mjs` and let `internal/releasesmoke/` call that one script rather than duplicating DOM selectors or asset-request checks across Go tests and GitHub workflow steps.
- Keep GitHub Actions Playwright setup on the hosted-runner browser download path (`bunx playwright install chromium`) instead of `--with-deps` unless the workflow truly owns OS package installation; live `apt` dependency resolution adds mirror/DNS flake outside the product behavior under test.
- When release automation needs cross-platform artifact smoke, package the full `dist/` once and fan out verification by downloading that shared artifact on each runner OS instead of rebuilding or re-describing the packaging contract in every matrix job.
- When tag-driven publication must wait for candidate smoke, keep the actual GitHub Release publish step in a follow-on workflow that resolves the same `v*` tag from the verified commit instead of racing a second tag-triggered publish workflow in parallel.
- Keep release-tag semver validation in one repository-owned helper and call it from both `make release` and GitHub Actions so malformed `v*` tags cannot slip into candidate or publish lanes through inconsistent regex snippets.
- Keep consumer install commands in `README.md` aligned with the repo-owned release surfaces: Homebrew should point at the generated cask tap, `curl ... install.sh | sh` should point at the hosted GitHub release asset, `go install` should match the supported module path, and manual archive fallback should remain available for constrained environments.
- When UI assets are committed for Go embedding, keep the build pipeline responsible for normalizing output filenames and refreshing any cache-busting stamp files instead of hand-editing `ui/dist/`.
- If repository Go builds fail in `ui/dist_stamp.go`, treat that file as part of the embedded-asset generation contract: resolve merge markers or regenerate it through the documented UI normalization flow instead of leaving `go test` blocked on a broken stamp file.
- When browser-backed UI replay tests and replay coverage reports share the same scenarios, keep that metadata in one repository-owned catalog so the tests, scripts, and docs cannot silently drift.
- When a UI lane requires browser evidence, record the exact Storybook/browser commands and the dashboard spot-check checklist in `docs/processes/manual-qa.md` so reviewers can reproduce the same local flow instead of relying on PR-only notes.
- When dashboard UI work needs reusable controls or states, add or extend primitives under `ui/src/components/ui/` first and keep feature folders focused on composition, data shaping, and product-specific behavior rather than bespoke chrome.
- When dashboard UI needs expandable sections with selection actions, pair `ui/src/components/ui/button.tsx` with `ui/src/components/ui/collapsible.tsx` and keep feature files responsible only for status-specific copy and token overrides rather than reintroducing raw disclosure button markup.
- When migrating dashboard charts away from bespoke SVG renderers, keep feature-owned sample shaping inside the feature module and route the actual rendering through `ui/src/components/ui/chart.tsx` plus Recharts so legend, tooltip, and loading treatment stay on the shared primitive path.
- When a dashboard surface needs a reusable read-only grid, build it on `ui/src/components/ui/table.tsx` or `ui/src/components/ui/data-table.tsx` and keep feature code responsible only for row shaping, captions, and product-specific cell content instead of hand-authoring new table chrome.
- When dashboard dialogs or forms render validation text, wire the affected controls with `aria-invalid` plus `aria-describedby` pointing at the active validation message so shared primitives satisfy the repository accessibility contract instead of relying on visual-only error copy.
- When GitHub Actions uses `actions/setup-go` against a module that declares a newer `toolchain` than its base `go` version, prefer uncached setup unless you have verified that cache restore does not collide with the auto-downloaded toolchain files in later jobs.
- When a cleanup lane closes with path or contract-alignment work, record the exact root-level verification commands in a `docs/development/*-closeout.md` artifact so the proof survives beyond `progress.txt`.
- When functional tests need `os.Chdir`, route the directory swap through the shared helper and serialize it with a package-level lock; process working directory is global state and will race any `t.Parallel()` coverage otherwise. If the same test changes directories more than once or does so inside subtests, use the scoped helper form so the lock is released between command executions instead of being held until parent-test cleanup.
- Keep `tests/functional_test/` focused on observable behavior. If a check only proves source layout, docs linkage, asset bundle contents, or command or route registration, move it to a narrower guard surface instead of adding it to the functional suite.
- Keep generated replay payload merge regressions in `pkg/replay/event_artifact_test.go` when the behavior under change is deterministic worker or workstation replacement and append ordering; seed `factoryapi.Factory` payloads directly there instead of rebuilding the same coverage through loaded-config filesystem fixtures.
- When a functional test exercises `WithRunAsync()` through `pkg/testutil/service_harness.go`, seed any required work before `RunInBackground()` and poll for the observable runtime state you need; fixed sleeps after starting an empty async loop can race a clean engine shutdown in CI.
- Keep user-facing CLI command examples in `README.md` aligned with `pkg/cli/root_test.go`; that test guards the documented `infinite-you docs ...` surface directly rather than inferring it from packaged docs internals.
- When a functional test covers `agent-factory docs`, run the command from a temp working directory without a local docs tree and assert user-visible headings or stable content markers rather than byte-for-byte parity with packaged markdown files.
- For current-factory functional coverage, use `tests/functional_test/named_factory_api_test.go` with the authored `service_simple` root fixture plus the generated Go client helpers; that path exercises `/factory/~current` from the default runtime before activation and from a named runtime after `POST /factory` switches the live service.
- When `make lint` fails in `cmd/deadcodecheck`, remove orphaned `_test.go` helper files and unused test wrappers before touching `docs/development/deadcode-baseline.txt`; the baseline is for accepted remaining debt, not abandoned local helpers.
- Use `pkg/api/openapi_contract_test.go` for narrow OpenAPI contract guards when the work is about authored schema structure or bundled route/schema alignment rather than handler runtime behavior.
- Keep the reserved default-runtime current-factory identifier documented in `api/components/schemas/data-models/FactoryName.yaml` and `/factory/~current`; regenerate bundled artifacts after authored OpenAPI edits instead of hand-patching `api/openapi.yaml` or generated clients.
- Keep Go OpenAPI client generation in `api/codegen_config/client.yaml` and `pkg/generatedclient/`; current-factory and named-factory functional helpers should call that generated client surface instead of maintaining handwritten HTTP request wrappers for those routes.
- When a generated Go client exists only to drive a bounded functional-test flow, constrain `api/codegen_config/client.yaml` with `include-operation-ids` for the exercised routes so the checked-in client does not duplicate unrelated contract models or require new handwritten-source guard exceptions.
- Keep named-factory activation semantics in `pkg/service/factory.go`; API handlers and higher-level tests should call that service seam so default-runtime fallback, durable-pointer writes, and runtime swaps stay under one owner.
- When current-factory reads need live runtime fallback data from `pkg/service/factory.go`, snapshot `runtimeCfg` through `currentRuntimeConfig()` before comparing directories or serializing it; `swapActiveRuntime` mutates that state under `runtimeMu`.
- Keep writable named-factory name validation centralized in `pkg/apisurface/`; create or import paths must reject the reserved `UNDEFINED` current-factory identifier there so handlers, services, and generated-client functional tests enforce the same public contract.
- Keep `docs/development/root-factory-artifact-contract-inventory.md` and `internal/testpath/artifact_contract.go` synchronized; the doc is not descriptive-only, it is a checked-in contract surface with order-sensitive tests.
- When public OpenAPI field names change, update `pkg/api/testdata/canonical-event-vocabulary-stream.json` together with the contract guards so fixture validation keeps exercising the current bundled vocabulary.
- When a live canonical event stream must close on subscriber cancellation, keep `pkg/factory/event_history.go` responsible for bridging producer fanout onto a subscriber-owned output channel; do not send directly to a channel that a cancellation path may close concurrently.
- When dispatch event identity ownership changes, update `api/components/schemas/events/FactoryEventContext.yaml`, the dispatch payload schemas, `pkg/factory/event_history.go`, and `pkg/api/testdata/canonical-event-vocabulary-stream.json` together so emitted events, generated contracts, and bundled vocabulary fixtures do not drift.
- When replay or projection code reconstructs dispatch chaining lineage, read `FactoryEvent.context.currentChainingTraceId` and `FactoryEvent.context.previousChainingTraceIds` first, then fall back to deprecated payload copies only for legacy events or fixtures.
- For end-to-end dispatch fan-in lineage changes, extend `tests/functional_test/factory_request_batch_test.go` so `TestFactoryRequestBatch_ChainingTraceFanIn_EndToEndSmoke` asserts the emitted dispatch event context, reconstructed world state, and workstation-request projection together instead of splitting the contract proof across helper-only tests.
- When dashboard timeline reconstruction or request-history projections rebuild dispatch chaining lineage, keep `ui/src/state/factoryTimelineStore.ts` aligned with the backend contract by reading `FactoryEvent.context.currentChainingTraceId` and `FactoryEvent.context.previousChainingTraceIds` first and treating dispatch-payload copies as legacy fallback only.
- When public request-batch field names change, update `pkg/interfaces/factory_runtime.go`, `pkg/factory/work_request_json.go`, and watched-file/worker batch fixtures together; those handwritten JSON boundaries are not generated and will drift silently if only `api/openapi.yaml` and generated clients are regenerated.
- Keep shared public worker and workstation enum aliases in `pkg/interfaces/public_factory_enums.go`; generated/OpenAPI boundary normalizers should call the strict helpers while generated/public round-trip helpers should call the permissive ones instead of cloning alias tables in `pkg/config` or UI export code.
- In `pkg/cli/dashboard`, derive completed and failed session work labels from `PlaceOccupancyWorkItemsByPlaceID` plus `PlaceCategoriesByID` first, and fall back to `Session.DispatchHistory` only when category-owned labels are absent so CLI summaries and failed-work details keep one compatibility path.
- In `pkg/config/factory_config_mapping.go`, keep mirrored generated-boundary retirement guards data-driven: define one retired-field inventory per boundary type and reuse it for both the top-level object and any nested `definition` object so only the caller-owned path text differs.
- `POST /work` is the single-submit surface and should reject relation graphs explicitly; named relation wiring belongs on `PUT /work-requests/{requestId}`, where the handler can resolve `sourceWorkName` and `targetWorkName` safely.
- `decodeSubmitWorkRequestBody` is the handwritten validation seam for `POST /work`; use the raw JSON field map there whenever generated structs need compatibility checks beyond Go's default `json.Unmarshal` behavior, especially when unary submit fields differ from canonical batch fields.
- For Redocly-bundled vendor extensions that must keep a schema pointer in `api/openapi.yaml`, store the extension value as a string JSON Pointer; nested `$ref` objects under `x-*` fields are inlined during bundling.
- When dashboard or world-view code needs runtime-only observability that is not reconstructed from canonical events, keep the event reducer unchanged and overlay the live engine snapshot fields at the projection boundary that already owns both inputs.
- Keep CLI dashboard dispatch-history fallback collection in `pkg/cli/dashboard/dashboard.go` behind one lane-owned helper that selects completed-versus-failed semantics centrally; future cleanup in this area should extend that shared seam instead of reintroducing parallel completed and failed wrapper owners.
- In `pkg/factory/runtime/dispatch_result_hook.go`, treat `CompletionDeliveryPlanner` presence as the only synchronous-dispatch mode switch; do not cache a second runtime boolean that can drift from `h.planner`.
- When `ui/dist_stamp.go` drifts or lands in conflict, regenerate it from checked-in `ui/dist/` with `bun ui/scripts/normalize-dist-output.mjs` instead of hand-merging hash values.
- For dispatcher pause-gating tests, use scheduler doubles that derive decisions from the `enabled` slice they receive so the assertions reflect the real FIFO/work-queue scheduler contract instead of depending on impossible post-selection decisions.
- When runtime guard or observability work needs a reusable reducer over normalized history, keep the pure derivation seam under `pkg/factory/internal/` and adapt dispatcher- or snapshot-specific inputs at the caller boundary instead of teaching the reducer about mutable subsystem state.
- When dispatcher throttle or guard work depends on failure timing, derive event time from `snapshot.DispatchHistory` and `interfaces.CompletedDispatch.EndTime`; `snapshot.Results` is a tick-local outcome surface and should not become the source of truth for pause timestamps.
- When extracting pure runtime reducers from stateful dispatcher logic, keep the reducer tests focused on semantic lane derivation and add dispatcher-level regression coverage for legacy stateful details such as preserving the original `PausedAt` when overlapping failures extend a live window.
- When subsystem behavior is derived from engine snapshots instead of local mutable caches, write multi-tick tests so later snapshots carry forward the owning runtime fields such as `DispatchHistory` and prior `ActiveThrottlePauses`; direct subsystem tests should mirror the engine snapshot contract rather than relying on hidden cross-tick state.
