## Codebase Patterns

- After a fresh `git pull`, verify recently changed candidate seams from live
  file reads before queuing work; the codebase-memory graph can lag `HEAD`
  briefly after merges.
- When generated public enum pointer helpers only differ by the output enum
  type, prefer one shared private nil-on-empty wrapper rather than repeating
  trim-check plus `&enumValue` plumbing across `pkg/interfaces`.
- When an API/service seam repeats an existing runtime interface method-for-method,
  prefer embedding the narrower interface instead of re-declaring the shared
  contract in a second package.
- Before treating files under `factory/inputs/**` or `factory/logs/meta/*` as
  checked-in workflow truth, confirm them with `git ls-files`; ignored local
  residue can coexist with the canonical control-plane surface.
- The checked-in meta control plane is `factory/logs/meta/asks.md`,
  `factory/logs/meta/view.md`, and `factory/logs/meta/progress.txt`; any other
  path should be treated as legacy or workspace-local until proven otherwise.
- Before queuing a new meta cleanup lane, compare the candidate file set
  against `gh pr view <number> --json files` for open PRs so the maintainer
  loop does not duplicate active work.
- When the root artifact inventory changes, keep
  `docs/development/root-factory-artifact-contract-inventory.md` and
  `internal/testpath/artifact_contract.go` aligned in the same change or the
  targeted artifact-contract test will fail immediately.
- Before re-queuing a local ignored cleanup idea under `factory/inputs/**`,
  compare it against recent merged PR history; ignored residue can remain in
  the workspace after its lane has already landed on `main`.
- Handwritten routes registered ahead of the generated API router can be
  intentional compatibility shims; verify whether they preserve more tolerant
  request parsing before treating them as removable duplication.
- When a handwritten compatibility route already normalizes tolerant query
  parsing before calling a handler, prefer keeping that behavior in the wrapper
  and removing any second copy of the same fallback logic from the handler.
- A package-local cleanup candidate can still collide with a broad open PR, so
  candidate validation must compare exact touched files against every active
  lane, not just against the candidate's apparent subsystem boundary.
- When a cleanup candidate lives in `pkg/workers`, inspect both script and
  inference/recording event payload assembly paths; small boundary-event helper
  duplication can span those sibling files even when the behavior is already
  test-covered.
- When a package-local cleanup candidate looks safe, verify the exact live PR
  file sets again after any sidecar exploration; open lanes can advance during
  the same maintainer pass and make an initially valid idea stale before it is
  queued.
- When the only open PR is a broad `tests/functional/**` reorganization, keep
  newly queued backend or contract lanes out of that tree and prefer package,
  contract, stress, or other non-colliding verification for the first cut.
- Before re-queuing a local cleanup idea, compare it against the most recent
  merged PR list as well as open PR file sets; fast-moving `main` can land the
  exact seam while the ignored workflow-input residue still suggests it is only
  queued.
- After a customer-ask implementation lane merges, re-check whether a legacy
  compatibility or fallback path still keeps the old abstraction live; the
  next highest-value cleanup is often removing that fallback rather than
  queueing an unrelated hygiene seam.
- Before treating tracked sibling `copy` scripts under `ui/scripts/` as live
  tooling, confirm the canonical entrypoints in `ui/package.json` and the
  checked-in development docs; duplicate script copies can be dead residue even
  when they sit beside actively used files.
- When a customer ask is already represented by an open PR, do not queue a
  second overlapping idea in the same file set; update the world model first,
  then keep parallel cleanup work confined to non-colliding seams.
- When a deprecated config field is already normalized away at the load
  boundary, prefer retiring any second in-memory runtime fallback for that
  alias instead of preserving duplicate compatibility in execution paths and
  test fixtures.
