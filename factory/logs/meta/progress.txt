## Codebase Patterns

- Validate repo-truth claims about `factory/inputs/**` with `git ls-files`
  before trusting visible local files; the repo tracks sentinels while live
  work items are ignored.
- Distinguish canonical queue state from helper planning residue: files under
  `factory/scripts/` can describe intended follow-up batches, but they do not
  satisfy the canonical watched submission surface under `factory/inputs/**`.
- Read `factory/logs/meta/asks.md` before submitting cleanup ideas; it is the
  canonical maintainer-routing surface and can explicitly forbid queue work.
- Refresh the meta model from live git state, not stale summaries; merge state,
  ignore rules, and local dirtiness can change within hours.
- Check `factory/factory.json` directly when summarizing the workflow because
  the live contract includes `to-complete`, `consume`, per-worktree execution,
  shared resource limits, cron cleanup, and loop breakers that replay fixtures
  may not reflect.
- Before queuing new `factory/inputs/**` work for an active ask, inspect ignored
  local residue in the canonical inboxes so the meta loop does not submit a
  duplicate batch over work that is already staged for execution.
- Reconcile ignored helper backlog files with open PRs and freshly merged
  branches before dispatching more cleanup work; local planning residue can be
  stale even when its scope still matches the ask surface.
- Before dispatching a fresh idea for a customer ask, inspect open PR bodies
  and changed-file ownership; an ignored local idea may already have advanced
  into an in-flight branch even if `main` has not merged it yet.
- Visible `factory/inputs/task/default/*.md` files can be stale ignored residue
  even after the owning PR has merged; re-check live PR state before treating a
  local task file as active work.
- Prune ignored local idea files once their owning PR has merged; otherwise the
  canonical inbox can retain stale residue that no longer represents an
  actionable backlog item.
- Import/export contract cleanup crosses more than one layer: confirm
  `api/openapi.yaml`, generated API artifacts, `pkg/config` mapping logic, and
  runtime/functional tests together before assuming a field has really been
  retired.
- Before queuing a narrow UI cleanup, verify the abstraction still has more
  than one production caller; if a wrapper has collapsed to a single caller, it
  is usually a better candidate for removal than preservation.
- Before preserving a handwritten API route in front of a generated handler,
  confirm whether it protects a real public contract or only a tolerant legacy
  parse path; if it only shields invalid legacy inputs, prefer deleting the
  shim and updating tests to assert the canonical route behavior.
- When the checked-in workflow docs require
  `factory/inputs/<work-type>/default/<file>`, treat acceptance of direct
  two-segment `factory/inputs/<work-type>/<file>` submissions as hidden legacy
  compatibility and a valid cleanup target.
- When the checked-in maintainer workflow and the public starter scaffold drift
  on inbox naming, pick one canonical path and prove the cleanup through
  init/bootstrap/run behavior instead of preserving both spellings in docs and
  smoke tests.
- When the default starter inbox contract changes, audit the generated
  `factory/inputs/README.md` template too; `pkg/cli/init` can keep stale
  starter-facing prose even after the runtime path, CLI help, and bootstrap
  tests are aligned.
- After a UI cleanup PR merges, re-read the exact live controls and chart
  contracts before closing the ask; copy reduction can land while residual
  iconography, branding, or axis-layout drift still remains.
- When replay timeline helpers carry the same legacy compat payload aliases or
  cast helpers in more than one file, consolidate them before adding new
  replay-path behavior so compatibility shims do not become a second API layer.
- After a broad replay-state dedupe PR merges, re-check whether one smaller
  projection helper still remains duplicated on the live path before declaring
  the whole timeline seam closed.
- When an authored runtime lookup path is already live, treat any remaining
  transition-ID fallback field or test fixture as a cleanup seam; prefer
  deleting the fallback and rewriting tests toward runtime-visible lane
  behavior instead of preserving dual ownership models.
- When a public JSON route contract moves from singular objects to arrays,
  inspect import/load boundary normalizers and checked-in fixture JSON next;
  hidden coercion can keep the retired authored shape alive after schemas,
  generated types, and UI consumers are already canonical.
- After a generated boundary and the main mapper both expose the same canonical
  field directly, look for post-processing compatibility patches that reparse
  the raw authored JSON again; those helpers are often dead seams that can be
  deleted with focused boundary/runtime regression coverage.
- For import/export portability asks, verify both `FlattenFactoryConfig` and
  expanded authored-layout writes before closing the lane; stripped fields in
  `WriteExpandedFactoryLayout` can still be rehydrated back into canonical
  `factory.json` export or runtime load paths elsewhere.
- When production request-shaping and functional-test request-shaping both
  exist, compare chaining-trace propagation and batch metadata field-by-field
  before assuming the duplicate helper is harmless; this repo already allowed a
  test-only copy to drift away from the live submit path.
- When a helper cluster only serves one long functional spec or one narrow test
  lane, prefer co-locating it with that spec over keeping a broad shared
  `helpers_test.go` owner; that reduces deadcode noise without weakening
  behavioral coverage.
- When a config-layer helper is only a pass-through to canonical
  `pkg/interfaces` enum conversion, delete the extra owner and keep mapping and
  layout callers on the shared helper instead of preserving duplicate wrapper
  surfaces.
- After a dedupe change merges, re-check the old owner package for alias-only
  wrappers; this repo can leave a one-function forwarding package behind even
  after the real behavioral duplication is gone.
- Re-check checked-in worldview notes against live `main` before dispatching a
  follow-up; merged cleanup PRs can invalidate the recorded “next seam” within
  a single refresh cycle.
- When an open PR touches a surface already treated as closed, compare the PR
  diff against the canonical ask before queueing sibling work; a new branch
  name can still hide overlap with an already-closed lane.
- Deadcode-baseline entries that only feed one long functional smoke are good
  candidates for the next narrow cleanup idea when live runtime, API, and CLI
  callers are absent.
- When deadcode-baseline entries still have live callers, compare the shared
  helper against suite-local copies before deleting it; duplicated test-helper
  ownership is often the real seam.
- When a test-helper file is split by build tags, check both the default and
  `functionallong` lanes before treating it as dead; replay-contract helpers
  can look unused in one lane while still serving the other.
- Treat graph or delegated explorer suggestions as provisional after a
  fast-forward; confirm the seam still exists on live `main` with `git log`,
  `rg`, and direct file reads before dispatching new cleanup work.
