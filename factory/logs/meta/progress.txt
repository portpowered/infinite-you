## Codebase Patterns

- Validate repo-truth claims about `factory/inputs/**` with `git ls-files`
  before trusting visible local files; the repo tracks sentinels while live
  work items are ignored.
- Distinguish canonical queue state from helper planning residue: files under
  `factory/scripts/` can describe intended follow-up batches, but they do not
  satisfy the canonical watched submission surface under `factory/inputs/**`.
- Read `factory/logs/meta/asks.md` before submitting cleanup ideas; it is the
  canonical maintainer-routing surface and can explicitly forbid queue work.
- Refresh the meta model from live git state, not stale summaries; merge state,
  ignore rules, and local dirtiness can change within hours.
- Check `factory/factory.json` directly when summarizing the workflow because
  the live contract includes `to-complete`, `consume`, per-worktree execution,
  shared resource limits, cron cleanup, and loop breakers that replay fixtures
  may not reflect.
- Before queuing new `factory/inputs/**` work for an active ask, inspect ignored
  local residue in the canonical inboxes so the meta loop does not submit a
  duplicate batch over work that is already staged for execution.
- Reconcile ignored helper backlog files with open PRs and freshly merged
  branches before dispatching more cleanup work; local planning residue can be
  stale even when its scope still matches the ask surface.
- Before dispatching a fresh idea for a customer ask, inspect open PR bodies
  and changed-file ownership; an ignored local idea may already have advanced
  into an in-flight branch even if `main` has not merged it yet.
- Prune ignored local idea files once their owning PR has merged; otherwise the
  canonical inbox can retain stale residue that no longer represents an
  actionable backlog item.
- Import/export contract cleanup crosses more than one layer: confirm
  `api/openapi.yaml`, generated API artifacts, `pkg/config` mapping logic, and
  runtime/functional tests together before assuming a field has really been
  retired.
- Before queuing a narrow UI cleanup, verify the abstraction still has more
  than one production caller; if a wrapper has collapsed to a single caller, it
  is usually a better candidate for removal than preservation.
- Before preserving a handwritten API route in front of a generated handler,
  confirm whether it protects a real public contract or only a tolerant legacy
  parse path; if it only shields invalid legacy inputs, prefer deleting the
  shim and updating tests to assert the canonical route behavior.
